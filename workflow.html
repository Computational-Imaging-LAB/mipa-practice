Pull Requests

Pull requests (PRs) are the heart of collaboration on GitHub. They let you tell others about changes you've pushed to a branch in a repository and request that someone review and pull in your contribution.

Creating a Pull Request

  1. Push your branch to GitHub: git push origin feature-branch
  2. Navigate to your repository on GitHub
  3. Click the "Pull requests" tab and then the "New pull request" button
  4. Select your branch as the "compare" branch
  5. Review your changes and click "Create pull request"
  6. Add a title and description that clearly explain your changes
  7. Click "Create pull request" to submit it

Anatomy of a Good Pull Request

  • Clear title: Summarize what the PR does
  • Detailed description: Explain why these changes are needed
  • Reference issues: Link to related issues with "#123"
  • Screenshots/videos: Include for UI changes
  • Test results: Show that your changes work as expected

Tip: Use keywords like "Fixes #123" or "Closes #123" in your PR description to automatically close the referenced issue when the PR is merged.

Code Reviews

Code reviews are a critical process where team members examine each other's code changes to improve quality and share knowledge.

Reviewing a Pull Request

  1. Navigate to the pull request on GitHub
  2. Click the "Files changed" tab to see the changes
  3. Comment on specific lines by clicking the "+" that appears when hovering
  4. Use suggestions to propose specific changes (click the suggestion icon)
  5. Review the entire PR by clicking "Review changes"
  6. Choose to comment, approve, or request changes
  7. Submit your review

Effective Code Review Practices

  • Be respectful and constructive
  • Focus on the code, not the person
  • Be specific about what needs changing
  • Explain why a change is needed, not just what
  • Look for both technical issues and design/architecture concerns
  • Suggest alternatives when possible
  • Remember to highlight good solutions too

Note: GitHub allows you to suggest exact code changes using the suggestion feature, which the author can directly apply with a single click.

Issues & Projects

GitHub's issue tracking and project management features help teams organize their work and track progress.

Working with Issues

Issues help you track bugs, features, and other tasks. Each issue can include:

  • Title and description
  • Assignees (who's responsible)
  • Labels (bug, enhancement, etc.)
  • Milestones (group issues by release)
  • Comments for discussion
  • Task lists with checkboxes

GitHub Projects

Projects provide Kanban-style boards for tracking work:

  • Project boards: Organize issues and PRs into columns
  • Automation: Automatically move cards based on status
  • Views: Table, board, and roadmap views
  • Custom fields: Add metadata to issues and PRs

Tip: Use issue templates to standardize how team members report bugs or request features, making it easier to collect necessary information.

Collaborative Branching Models

Teams need a structured approach to branches to maintain order in collaborative development.

GitHub Flow

A lightweight, branch-based workflow:

  1. Create a branch from main
  2. Add commits
  3. Open a pull request
  4. Discuss and review code
  5. Deploy (optionally) for testing
  6. Merge to main

Git Flow

A more complex model with specific branch types:

  • main: Always production-ready code
  • develop: Latest development changes
  • feature/: New features
  • release/: Preparing for a release
  • hotfix/: Urgent production fixes

Trunk-Based Development

A model focused on keeping the main branch clean:

  • Developers work on short-lived feature branches
  • Frequent integration to main (at least daily)
  • Emphasis on automated testing
  • Feature flags for incomplete features

Note: Choose a branching model that fits your team's size, release cadence, and workflow preferences. Simpler is often better for smaller teams.

Handling Merge Conflicts

Merge conflicts occur when two branches have made different changes to the same line of a file, or when one branch deletes a file that another branch has modified.

Resolving Merge Conflicts

When a conflict occurs during a merge or pull request:

  1. Git will mark the conflicted areas in your files
  2. Open the conflicted files and look for conflict markers (<<<<<<<, =======, >>>>>>>)
  3. Edit the files to resolve the conflicts
  4. Remove the conflict markers
  5. Add the resolved files: git add <filename>
  6. Complete the merge with git commit

Avoiding Merge Conflicts

  • Pull from the main branch frequently
  • Keep feature branches short-lived
  • Communicate with team members about overlapping work
  • Use smaller, focused commits
  • Structure your project to minimize file overlap

Tip: GitHub has a web-based editor for resolving simple conflicts directly on the website. For more complex conflicts, use a visual merge tool like Visual Studio Code, GitKraken, or SourceTree.
